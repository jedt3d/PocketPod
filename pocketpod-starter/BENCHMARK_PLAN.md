# PocketPod Benchmark Plan

## Serverpod Baseline

PocketPod benchmarks are built on [Serverpod](https://serverpod.dev). The current copied Serverpod source baseline is:

```text
3.5.0-beta.10
```

PocketPod release tags match the Serverpod baseline version, such as `v3.5.0-beta.10`, so benchmark results can be tied back to the exact Serverpod source version in `serverpod-pocketpod`.

## Purpose

Before Phase 3, measure whether the SQLite tuning in Phase 2 gives useful evidence, not just correct PRAGMA values.

This benchmark is not a final production capacity claim. It is a local engineering gate to compare configurations, catch obvious lock contention, and decide whether the Serverpod + SQLite direction is still worth extending with admin tooling.

## Comparison Targets

Primary comparison:
1. `serverpod-sqlite-tuned`: current Phase 2 configuration, using WAL, `synchronous=NORMAL`, `busy_timeout=5000`, and `maxConnectionCount: 10`.
2. `serverpod-sqlite-untuned`: same app with SQLite tuning disabled or overridden to a lower-concurrency baseline.

Optional comparison:
3. `serverpod-postgres-default`: stock Serverpod/Postgres scaffold, only if we need a framework baseline.
4. `pocketbase-local`: PocketBase on the same machine, only if the goal is a PocketBase-style performance comparison.
5. `direct-sqlite-dart`: a minimal Dart `sqlite_async` harness with no Serverpod HTTP/RPC layer, to isolate framework overhead.

Ambiguity:
If "Codex-based performance" means a Codex-generated direct SQLite service, use `direct-sqlite-dart`. If it means PocketBase-style performance, use `pocketbase-local`. If it means Serverpod code generated by Codex, the primary tuned vs untuned comparison is the right baseline.

## Workloads

Use deterministic local workloads with a fresh database per run.

1. Smoke:
   - Start server.
   - Apply migrations.
   - Confirm `PRAGMA journal_mode`, `PRAGMA synchronous`, and `PRAGMA busy_timeout`.

2. Read-only:
   - Seed 10,000 rows.
   - Run concurrent list/get requests.
   - Measure requests/sec, p50, p95, p99 latency, errors.

3. Write-only:
   - Run concurrent create requests.
   - Measure successful writes/sec, p50, p95, p99 latency, lock/busy errors.

4. Mixed workload:
   - 80% reads, 20% writes.
   - Run at concurrency levels 1, 5, 10, 25, 50.
   - Measure throughput, latency, error rate, database file growth.

5. Burst contention:
   - Fire simultaneous writes at concurrency 10, 25, 50.
   - Confirm busy timeout causes waiting rather than immediate lock failure.

## Metrics

Record:
- Runtime target and git SHA.
- Machine model, CPU count, RAM, OS.
- Dart and Flutter versions.
- Serverpod package source and version.
- SQLite database path.
- PRAGMA values.
- Concurrency level.
- Total requests.
- Successful requests/sec.
- p50, p95, p99 latency.
- Error count and error classes.
- Database file, `-wal`, and `-shm` sizes after run.

## Harness Plan

Add a `tool/benchmarks/` harness in PocketPod:

- `tool/benchmarks/README.md`: how to run benchmarks and interpret results.
- `tool/benchmarks/run_bench.dart`: orchestrates seed, workload, and result capture.
- `tool/benchmarks/results/*.json`: machine-readable output, ignored by default.
- `tool/benchmarks/results/*.md`: checked-in summary only when intentionally publishing a benchmark result.

Use generated Serverpod client calls when possible. If client overhead blocks useful measurements, add a minimal benchmark endpoint and call HTTP/RPC directly.

## Acceptance Gate Before Phase 3

Do not start Phase 3 until we have at least:

1. Tuned vs untuned comparison for read-only, write-only, mixed, and burst workloads.
2. Zero unexpected lock errors under concurrency 10 mixed workload.
3. A clear p95 latency and throughput table in `MILESTONE.md`.
4. A written decision:
   - continue to Phase 3,
   - tune further,
   - change Serverpod/SQLite approach,
   - or compare against PocketBase/Postgres before proceeding.

## Stop Rules

Stop and report before Phase 3 if:
- tuned SQLite still has lock failures at low concurrency,
- Serverpod startup/migration behavior is unstable,
- benchmark results vary too much to interpret,
- direct SQLite is strong but Serverpod overhead dominates,
- PocketBase comparison becomes necessary for the product decision.
